# Экспорт PKCS#12

PKCS#12 (`.p12` / `.pfx`) объединяет сертификат конечного субъекта,
закрытый ключ и полную цепочку УЦ в один файл, защищённый паролем.
Этот формат принимают хранилища сертификатов Windows, браузеры и
большинство VPN/почтовых клиентов.

## Когда использовать PKCS#12

* Импорт в хранилище сертификатов Windows (`certmgr`, `certlm`).
* Настройка почтового клиента (Thunderbird, Outlook) для S/MIME.
* Подготовка VPN-клиента, которому нужен единый файл с ключом.
* Любой инструмент, не принимающий отдельные PEM-файлы сертификата,
  ключа и цепочки.

## Экспорт

На странице деталей сертификата (`/certificates/<uuid>/`) нажмите
**Export PKCS#12**. Под панелью кнопок откроется встроенная форма с
полем пароля; нажатие **Download .p12** скачает файл.

Кнопка показывается только если у сертификата есть закрытый ключ,
хранящийся на сервере. Если сертификат был выпущен по внешнему CSR,
ключ на стороне OwnCA не сохраняется и кнопка скрыта — собирать
просто нечего.

**Требования к паролю:**

* Не может быть пустым — OwnCA отклоняет пустой пароль.
* На уровне браузера установлен минимум в 4 символа
  (`minlength="4"`); максимум приложение не ограничивает. Следуйте
  политике паролей вашей организации.
* Тот же пароль потребуется при импорте файла.

## Содержимое бандла

Экспортированный `.p12` включает:

| Bag | Содержимое |
|---|---|
| `certBag` | Сертификат конечного субъекта (friendly name = Common Name). |
| `keyBag` | Закрытый ключ. |
| `certBag` × N | Полная цепочка УЦ — промежуточные + корневой, собранная по полю издателя. |

## TK-26 совместимый вариант (только ГОСТ-ключи)

Если у сертификата ГОСТ-ключ (ГОСТ Р 34.10-2012 256 или 512 бит),
в одной строке с полем пароля появляются дополнительный флажок и
селектор шифра:

> [x] **TK-26 совместимый формат (GOST PFX)**
> Шифр: [`kuznyechik-ctr-acpkm` ▾]

Что включает флажок: PFX собирается по RFC 9337 + RFC 9548 (TK-26).
Keybag и cert envelope шифруются PBES2 с PBKDF2-HMAC-Streebog поверх
выбранного CTR-ACPKM-шифра (Кузнечик или Магма); внешний MAC —
HMAC-Streebog-512 c KDF из RFC 9548 §3 (PBKDF2 с `dkLen=96`,
последние 32 октета — ключ HMAC). Эту схему понимает CryptoPro CSP
и другие TK-26-совместимые криптопровайдеры.

Селектор задаёт одно значение для `-keypbe` и `-certpbe` openssl
pkcs12 (одинаковый шифр для keybag и cert envelope). Доступны
`kuznyechik-ctr-acpkm` (по умолчанию) и `magma-ctr-acpkm`.

Для ГОСТ-сертификата существует **два** валидных PFX — выбирайте по
целевому потребителю:

* RSA совместимые криптопровайдеры — снимите флажок. Получите
  стандартный `.p12` (PBES2 / AES-256-CBC / PBKDF2-HMAC-SHA-256).
* TK-26 совместимые криптопровайдеры — оставьте флажок включённым,
  при необходимости поменяйте шифры. Файл `.gost.p12` импортируется
  без преобразований.

Если флажок выставлен для RSA-сертификата, dashboard отклонит запрос
с ошибкой — формат не имеет смысла без ГОСТ-ключа.

Реализация: на стороне сервера запускается стандартный
`openssl pkcs12 -export` против gost-engine с поддержкой
RFC 9337/9548
([`gost-engine/engine`](https://github.com/gost-engine/engine),
ветка `master`; влито в апстрим в PR #527).

## Примеры импорта

**Windows:**
```
certutil -importpfx -p "пароль" cert.p12
```

**macOS:**
```
security import cert.p12 -P пароль -k login.keychain
```

**Linux (p11-kit / NSS):**
```
pk12util -i cert.p12 -d sql:$HOME/.pki/nssdb -W пароль
```

**TK-26 совместимый криптопровайдер под Windows (`.gost.p12`):**

```
csptest -keyset -container "\\.\HDIMAGE\<имя-контейнера>" \
        -keytype exchange -import cert.gost.p12 -pwd "пароль"
```

…либо через GUI: *Крипто-Про → Сервис → Установить личный
сертификат* и указать файл `.gost.p12`.
